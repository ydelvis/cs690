# AWS Lambda Permission Inference — CS690

Static + LLM-assisted inference of least-privilege IAM permissions for serverless
Lambda functions, with evaluation against a static oracle ground truth.

## Installation

### 1. Python dependencies

```bash
pip install \
  pyyaml \
  requests \
  pandas \
  numpy \
  faiss-cpu \
  sentence-transformers \
  tree-sitter==0.20.4 \
  python-dotenv          # optional, for .env file parsing
```

> **faiss-gpu**: if you have a CUDA GPU, replace `faiss-cpu` with `faiss-gpu`.

### 2. Tree-sitter grammar repos (for JS and Go call graphs)

Clone the grammar repos **into the project root**:

```bash
git clone https://github.com/tree-sitter/tree-sitter-javascript
git clone https://github.com/tree-sitter/tree-sitter-go
# Python call graphs use stdlib ast — no grammar repo needed
```

The first run of `build_call_graphs.py` or `extract_aws_calls.py` will compile
`build/my-languages.so` automatically from whichever repos are present.
You can skip Go or JavaScript by simply not cloning that repo; those files will
be skipped with a warning.

### 3. Ollama (for Steps 6 and 7)

```bash
# Install Ollama, then pull the model
ollama pull qwen3-coder:30b
```

Ollama must be reachable at the URL set in `OLLAMA_URL` (see Configuration).

## Configuration

All configurable values are constants near the top of each script.
Edit them directly before running.

### `src/rag_pipeline.py` and `src/llm_validate_permissions.py`

| Constant | Default | Description |
|---|---|---|
| `OLLAMA_URL` | `http://localhost:11434/api/generate` | **Required.** Full URL to your Ollama instance. |
| `OLLAMA_MODEL` | `qwen3.5:27b` | Model name as it appears in `ollama list`. |
| `EMBED_MODEL` | `BAAI/bge-base-en` | Sentence-transformer embedding model *(rag_pipeline.py only)*. |
| `TOP_K` | `5` | Base number of chunks retrieved per query *(rag_pipeline.py only)*. Total retrieved = `TOP_K × (n_permissions + 1)`. |

## Pipeline: Step-by-Step

### Step 1 — Build the permission database

Compresses `dataset/database/` into a single queryable JSON file.
This only needs to run **once**; the output is static.

```bash
python3 src/build_permission_db.py
```

**Output:** `dataset/permissions.json`

Each entry maps an AWS SDK action to its base permissions and which
optional parameters trigger additional permissions:

```json
"s3.copyObject": {
  "base_permissions": ["s3:GetObject", "s3:PutObject"],
  "param_permissions": {
    "ObjectLockLegalHoldStatus": ["s3:PutObjectLegalHold"],
    "ObjectLockMode":            ["s3:PutObjectRetention"]
  },
  "no_impact_params": ["BucketKeyEnabled", "CacheControl", ...],
  "ambiguous_params": ["TaggingDirective"],
  "sample_count": 12
}
```

---

### Step 2 — Extract app configurations

Parses every `serverless.yml` (and any `.env` files) in `dataset/apps/`.
Produces a normalised JSON with environment variables, function handlers,
and IAM statements declared in the config.

```bash
python3 src/extract_configs.py
```

**Output:** `configs/<app-name>.json`  
**Requires:** `pyyaml` (`pip install pyyaml`)

---

### Step 3 — Build call graphs

Performs static call-graph analysis on all source files (Python via `ast`,
JavaScript and Go via tree-sitter). Produces per-app adjacency maps used
to trace which functions are reachable from each Lambda entry point.

```bash
python3 src/build_call_graphs.py
```

**Output:** `call_graphs/<app-name>.json`  
**Requires:** tree-sitter grammar repos cloned (Step 3 of Installation)

> Steps 2 and 3 can run in either order and are independent of each other.

---

### Step 4 — Extract AWS SDK calls

For each Lambda entry function (as declared in `serverless.yml`), traces
the call graph and finds every AWS SDK call reachable from that entry point.
Extracts parameter names and resolves static values from literals and env vars.

```bash
python3 src/extract_aws_calls.py
```

**Output:** `aws_calls/<app-name>.json`  
**Depends on:** Steps 2 and 3 (configs and call graphs must exist)

Supported patterns:

| Language | Pattern |
|---|---|
| Python | `boto3.client('s3').put_object(...)`, `client = boto3.client('s3'); client.get_object(...)` |
| JavaScript | `new AWS.S3(); s3.putObject({...})`, `const db = require('./dynamodb'); db.put({...})` |
| Go | `&s3.CopyObjectInput{Bucket: ..., Key: ...}` struct literals |

---

### Step 5 — Infer static permissions

Maps each extracted AWS call to IAM permissions using `dataset/permissions.json`.
Produces two permission sets per function:

- **`baseline_permissions`** — derived from the method name alone (no param info).
  This is what you get from knowing only *which* SDK methods were called.
- **`full_static_permissions`** — derived from method name + observed parameters.
  This is the oracle / ground truth used for evaluation.

```bash
python3 src/infer_static_permissions.py
```

**Output:** `static_permissions/<app-name>.json`  
**Depends on:** Steps 1 and 4

---

### Step 6 — LLM validation *(requires Ollama)*

Takes the baseline permissions and the application's source code (handler file +
one-level imports), and asks the LLM to validate, add, or remove permissions.

**Before running**, set `OLLAMA_URL` and `OLLAMA_MODEL` in `src/llm_validate_permissions.py`.

```bash
python3 src/llm_validate_permissions.py
```

**Output:** `llm_permissions/<app-name>.json`  
**Depends on:** Step 5  
**Requires:** Ollama running and reachable

---

### Step 6b — RAG-assisted LLM validation *(requires Ollama + sentence-transformers + faiss)*

Alternative to Step 6. Instead of injecting the raw handler file, builds a
FAISS vector index over all source functions in the app and retrieves the most
semantically relevant code chunks for each permission. Scales better to large
apps and finds helpers in files not directly imported by the handler.

Retrieval is hybrid:
- **Semantic**: one query per baseline permission + one for the handler function
- **Call-graph**: functions reachable from the entry point are always included

**Before running**, set `OLLAMA_URL` and `OLLAMA_MODEL` in `src/rag_pipeline.py`.

```bash
python3 src/rag_pipeline.py
```

**Output:** `rag_permissions/<app-name>.json`  
**Depends on:** Steps 3 and 5 (call graphs and static permissions must exist)  
**Requires:** Ollama + `pip install sentence-transformers faiss-cpu`  
**Cache:** FAISS indexes are cached in `.cache/` (keyed by app path). Delete `.cache/` to force rebuild.

Steps 6 and 6b are independent and can run in either order or in parallel. Run both to get a three-way comparison in Step 7.

---

### Step 7 — Evaluate

Compares baseline, LLM-assisted, and RAG-assisted permissions against the ground
truth (`full_static_permissions`). Computes Precision, Recall, and F1 per function
and macro-averaged across all functions. Whichever of Steps 6 / 6b have been run
will appear in the report; missing output directories are skipped gracefully.

```bash
python3 src/evaluate_permissions.py
```

**Output:**
- `evaluation/results.json` — per-function details (true/false positives/negatives)
- `evaluation/summary.json` — aggregate metrics by method and by app

**Depends on:** Step 5 (and optionally Step 6 for LLM comparison)

## Running the Full Pipeline

```bash
# From the project root: cs690/

# One-time setup
python3 src/build_permission_db.py       # Step 1

# Per-run (re-run whenever apps change)
python3 src/extract_configs.py           # Step 2
python3 src/build_call_graphs.py         # Step 3
python3 src/extract_aws_calls.py         # Step 4
python3 src/infer_static_permissions.py  # Step 5

# LLM steps (require Ollama — see Configuration; run one or both)
python3 src/llm_validate_permissions.py  # Step 6   — raw source injection
python3 src/rag_pipeline.py             # Step 6b  — RAG retrieval (independent of Step 6)

# Evaluation (scores whichever of Steps 6 / 6b have been run)
python3 src/evaluate_permissions.py      # Step 7
```

Steps 2 and 3 can run in parallel. All other steps must run in order.

## Troubleshooting

### `build/my-languages.so` errors

```
RuntimeError: No tree-sitter grammar repos found.
```

Clone at least one grammar repo into the project root (see Installation §2).
JS and Go are optional — Python call graphs always work via stdlib `ast`.

### `YAML parse error` on `serverless.yml`

Usually caused by CloudFormation tags (`!Ref`, `!Sub`). These are handled
automatically; other malformed YAML will be skipped with a warning and that
app's config will be empty.

### Ollama connection refused

- Confirm Ollama is running: `ollama serve`
- Check `OLLAMA_URL` points to the correct host and port (default `11434`)
- Confirm the model is pulled: `ollama list`

### `permissions.json` lookup returns empty

The method name may not be in the database. Check the action key format:
`<service>.<camelCaseMethod>` (e.g., `dynamodb.putItem`, `s3.copyObject`).
Python boto3 snake_case names (`put_item`) are converted automatically by
`extract_aws_calls.py`.

### Apps showing 0 AWS calls

Common causes:
1. The handler string in `serverless.yml` doesn't resolve to an existing file — check `configs/<app>.json` to confirm `handler` values parsed correctly.
2. The app uses an unsupported SDK pattern (e.g., SDK v3 `@aws-sdk/client-s3` for JS).
3. For Go apps: the grammar repo is missing, or `lambda.Start()` uses a wrapper not yet recognised.
